# 苏格拉底对话记录 — 第二次（2026年4月）

*主题：平台数据设计、社区建设、MVP定义*

---

## 对话摘要

### 起点：自选场景的漏洞

**问题**：如果学习者可以自选场景来展示"迁移应用"能力，他们会不会选最简单的场景来应付守门？

**Simon 的回答**：一定程度上靠自觉，但也可以容忍。

> "水至清则无鱼——先把社区建起来，之后再加防弹机制。"

**引出的原则**：
- 平台建设初期，降低门槛比严格守门更重要
- 防作弊机制是第二阶段的事，不是第一阶段的设计约束
- 但**研究数据的设计**不能等到第二阶段——需要从一开始就记录下"学生选择了什么难度的场景"这个数据点

---

### 第一个关键决策：运营数据即研究数据

**问题**：平台运营数据和学术研究数据需要分开收集吗？

**Simon 的答案**：不用分开，应该设计成同一套数据直接可用于研究。

> "最好当然是平台运营数据直接可以用来做研究，而且我们甚至可以把这些数据匿名开放，让更多人可以在上面做研究。"

**引出的设计原则**：
- 平台从第一天起就是一个"活的研究数据库"
- 匿名数据集开放 = 吸引外部研究者，扩大学术影响力
- 类比：Mozilla Common Voice、OpenAlex、Galaxy Zoo 的开放数据模式

**对论文的影响**：
- Paper B（xAPI数据分析）的数据集可以开放，变成一个 public dataset
- 这个公开数据集本身就可以发一篇 data paper（第五篇论文的可能性）

---

### 第二个关键决策：数据所有权与知情同意

**问题**：学习者知道自己的数据会被用于研究、对外开放吗？

**Simon 的答案**：一开始就告诉用户，管理好预期。

> "一开始就要给大家管理一个预期——你不要讲一些很私人的东西，你做一个学习的过程，你把数据贡献出来，这不是很正常啊。"

**引出的设计原则**：
- 注册时明确声明：学习数据（匿名）将用于教育研究
- 参照 LLM 厂商的数据政策类比，用户对"学习数据被使用"有一定预期
- 用语要正面积极："你的数据在帮助建设更好的学习科学"

**对同意书设计的影响**：
- 现有同意书模板（`consent-forms/`）需要加一节"学习行为数据用途声明"
- 声明需要区分：原始对话（不开放）vs. 匿名行为数据（可开放）

---

### 第三个关键决策：BYOK 用户也收 xAPI 数据

**问题**：自带 API 密钥的高水平用户，他们的数据收不到，研究样本是否系统性偏向初学者？

**Simon 的答案**：不是的——BYOK 用户的学习行为依然需要收集。

> "即便他自己带 API 密钥，我还是得收集一定的数据，因为我不收数据的话，我怎么能够分析他的学习成果，我怎么能给他证书呢？"

**引出的架构决策**：

| 数据类型 | 平台 AI 用户 | BYOK 用户 |
|---------|------------|---------|
| xAPI 学习行为 | ✅ 收集 | ✅ 收集（证书所需） |
| AI 对话原文 | ✅ 平台可见 | ⚠️ 走用户自己的 API，平台不存储 |
| 研究数据集 | ✅ 可纳入 | ✅ 行为部分可纳入 |

**结论**：采样偏差问题不存在——xAPI 对所有用户都是 always on。

---

### 第四个关键决策：MVP 定义

**问题**："把平台搭起来"的最小版本是什么？

**Simon 的答案**：

> "只要搞一两篇文章，能够把整个流程跑通就可以了。一旦跑通了，就意味着有一个模板，剩下的就比较容易了，可以让其他人接在这个技术上做。"

**MVP 的精确定义**：

```
1 篇 ProComm 文章（内容已有）
    ↓
学习者打开互动页面，阅读文章
    ↓
与 AI 对话，展示"迁移应用"（自选场景）
    ↓
xAPI 记录全程行为
    ↓
AI 初评 → 达到门槛 → 系统发出证书
    ↓
✅ 全流程跑通 = MVP 完成
```

**证书技术**：
- 初期：PDF 证书 + 唯一码（简单可靠）
- 中期：Open Badge 标准（IEEE 背书增加说服力）
- 可探索：区块链验真（更多说服力，复杂度高，暂不优先）

**Simon 的核心判断**：
> "我们主要是这个协会本身就名气很大，很有说服力，所以我们只要把代码写得比较鲁棒，这个就应该可以没什么问题。"

**IEEE 品牌 = 最强的信任机制**，不需要靠区块链来背书，鲁棒的代码 + IEEE 签名就够了。
