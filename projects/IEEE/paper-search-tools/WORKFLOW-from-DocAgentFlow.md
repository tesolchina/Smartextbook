# DocAI 开发工作流程 — Development Workflow

> 最后更新：2026-04-11 (v2.79.2)
> 目标：即使只有一个人 + AI，也要像专业团队一样有条理地工作

---

## 核心原则

1. **发现问题和修复问题是两个独立动作** — 发现时记录，不一定马上修
2. **每次只做一件事** — 不要修 bug 的时候顺便加新功能
3. **先测试再发布** — 没有通过回归测试的代码不上线
4. **记录一切** — 未来的你会感谢现在的你

---

## 一、发现 Bug 的流程

```
发现问题 → 记录 → 评级 → 排入队列 → 修复 → 测试 → 关闭
  (1分钟)   (5分钟)  (1分钟)  (按优先级)  (不定)  (必须)  (更新文档)
```

### 第1步：发现时立刻记录（1分钟）

不管你在做什么，**先停下来记录**，不要去追问题的根源。
告诉 AI：

> "我发现了一个问题：[描述现象]。请帮我记录到 docs/testDebugs/bugs/INDEX.md，不需要修复。"

AI 会在 INDEX.md 的 Open 列表中添加一条，包含：
- 简要描述
- 严重度初判（P0/P1/P2）
- 发现日期

### 第2步：评级（AI 协助）

| 严重度 | 标准 | 处理时间 |
|--------|------|---------|
| **P0** | 用户无法使用核心功能（聊天崩溃、登录失败、数据丢失） | 24小时内修复 |
| **P1** | 功能异常但有替代方案（特定工具不工作、显示错误） | 本周处理 |
| **P2** | 体验不佳但不影响使用（UI 细节、性能慢、功能缺失） | 下次迭代 |

### 第3步：决定何时修复

- **P0**：立即放下手头工作去修
- **P1**：记录后继续当前工作，在下一个工作时段开始前处理
- **P2**：放入待办，攒到一定数量一起处理

### 第4步：修复流程

告诉 AI：

> "请修复 O[编号]：[问题描述]。修复后运行回归测试。"

AI 完成后会：
1. 修复代码
2. 运行 `npx tsx tests/regression.ts`（必须全部通过）
3. 更新 INDEX.md 状态（从 Open 移到 Fixed）
4. 更新 changelog.md
5. 更新版本号

### 第5步：验证和关闭

修复后，在生产环境验证：
- 如果OK → bug 正式关闭
- 如果还有问题 → 重新打开，补充新发现

## 二、新功能开发流程

```
想法 → 调研 → 设计 → 实现 → 测试 → 发布
(记录)  (1天)  (半天)  (不定)  (必须)  (验证后)
```

### 第1步：记录想法

告诉 AI：

> "我有一个新想法：[描述]。请帮我记录到 content/docs/research/ 文件夹，暂时不需要实现。"

这样想法不会丢失，也不会打断当前工作。

### 第2步：调研（AI 扮演产品经理）

告诉 AI：

> "请帮我调研 [功能名称]，包括：
> 1. 这个功能解决什么问题？
> 2. 类似产品怎么做的？
> 3. 我们实现的技术难度和工作量？
> 4. 建议的实现方案？
> 请写到 content/docs/research/[功能名].md"

### 第3步：设计（AI 扮演架构师）

调研完成、决定要做之后：

> "请为 [功能名称] 做技术设计：
> 1. 需要改哪些文件？
> 2. 数据模型变化？
> 3. API 接口设计？
> 4. 前端页面/组件？
> 5. 可能影响的现有功能？
> 不要写代码，只写设计文档。"

### 第4步：实现

> "请按照 content/docs/research/[功能名].md 的设计方案实现 [功能名称]。"

实现时注意：
- 一次只实现一个功能
- 每完成一个功能点就运行回归测试
- 如果发现新 bug，记录但不在这次修复（除非是 P0）

### 第5步：测试

实现完成后，必须通过：
1. `npx tsx tests/regression.ts` — 回归测试（~5秒）
2. 手动在浏览器中验证新功能
3. 如果涉及 AI 行为 → `npx tsx tests/integration.ts`（~5分钟）

### 第6步：发布

> "请更新版本号和 changelog，然后构建发布。"

## 三、日常工作建议

### 每次开始工作时

告诉 AI：

> "请给我当前项目状态报告：
> 1. 有哪些未解决的 P0/P1 bug？
> 2. 正在进行的功能开发？
> 3. 上次工作未完成的事项？"

### 工作时段的结构

```
[开始] → 检查状态 → 处理 P0 bug → 继续进行中的工作 → 记录新发现 → [结束]
                         ↑                                    |
                         └── 如果发现新 P0 ──────────────────┘
```

**建议的时间分配：**
- 70% 时间：修复 bug + 稳定现有功能（当前阶段）
- 20% 时间：调研和设计（为下阶段准备）
- 10% 时间：记录和整理文档

### 每次结束工作时

告诉 AI：

> "请更新项目状态：
> 1. 今天修复了什么？
> 2. 发现了什么新问题？
> 3. 下次工作应该从哪里开始？
> 请更新 docs/testDebugs/bugs/INDEX.md 和 content/docs/changelog.md"

## 四、AI 角色分工

虽然你是一个人，但你可以让 AI 在不同阶段扮演不同角色：

| 你想做的事 | 告诉 AI 的角色 | 示例指令 |
|-----------|--------------|---------|
| 发现并记录问题 | QA 测试员 | "请以 QA 角色检查 [功能]，记录所有问题" |
| 分析问题原因 | 调试工程师 | "请分析这个错误的根本原因" |
| 设计解决方案 | 架构师 | "请设计修复方案，不要直接写代码" |
| 实现修复 | 开发工程师 | "请按方案修复，完成后跑回归测试" |
| 调研新功能 | 产品经理 | "请调研 [功能]，写分析报告" |
| 代码审查 | 审查员 | "请检查最近的改动有没有问题" |

## 五、自动化测试策略

> **测试入口（Agent 必读）**：在运行任何测试或准备发布之前，请先阅读
> **[`docs/testDebugs/testGo.md`](testDebugs/testGo.md)**。
> 这是一个自包含的 runbook，涵盖所有测试阶段（A–D）、发布前步骤和发布后验证，
> 无需其他额外上下文即可独立执行。下方表格是快速参考；完整决策逻辑请以 testGo.md 为准。

### 目前有的测试

| 测试套件 | 测试数 | 耗时 | 费用 | 运行时机 |
|---------|--------|------|------|---------|
| regression.ts | 133 | ~5s | $0 | 每次改代码后 |
| core-tools-regression.ts | ~50 | ~10s | $0 | 改工具相关代码后 |
| build-verification.ts | ~56 | ~3s | $0 | 发布前 |
| feishu-regression.ts | ~30 | ~3s | $0 | 改飞书功能后 |
| integration.ts | ~27 | ~5min | ~$0.15 | 改 AI 行为后 |
| e2e-ui-audit.ts | ~15 | ~30s | $0 | 改 UI 后 |

### 需要补充的测试（按优先级）

**立即可做（不花钱）：**
1. Tab URL 解析测试 — 验证 Google Doc tab 参数不被丢弃
2. 模型回退链测试 — 模拟主模型失败时的错误处理
3. Token 估算准确性测试 — 固定输入验证估算值合理

**需要 AI 调用（花少量钱）：**
4. 搜索触发条件测试 — 简单请求不应触发搜索
5. 长会话上下文测试 — 历史压缩后关键指令是否被遵循

### 什么时候跑什么测试

```
改了后端代码 → regression.ts（必须）
改了工具定义 → regression.ts + core-tools-regression.ts
改了 AI prompt → regression.ts + integration.ts
改了前端 UI → regression.ts + e2e-ui-audit.ts
改了飞书相关 → regression.ts + feishu-regression.ts
准备发布 → 全部测试
```

## 六、文件夹用途说明

| 文件夹 | 用途 | 什么时候写 |
|--------|------|-----------|
| `docs/testDebugs/bugs/` | Bug 报告和追踪 | 发现 bug 时 |
| `docs/testDebugs/bugs/INDEX.md` | Bug 总索引（状态看板） | 每次 bug 状态变化时 |
| `content/docs/research/` | 调研报告 | 考虑新功能前 |
| `content/docs/changelog.md` | 版本变更记录 | 每次发版时 |
| `docs/quality/` | 测试计划和质量报告 | 改测试策略时 |
| `docs/architecture/` | 技术架构文档 | 重大架构变化时 |
| `docs/product/` | 产品功能文档 | 新功能上线后 |
| `docs/business/` | 商业计划 | 战略规划时 |
| `.local/tasks/` | 系统任务文件 | 由 AI 自动管理 |

## 七、版本号规范

```
v2.主版本.补丁版本

主版本 +1：新功能上线（如 v2.79 → v2.80）
补丁版本 +1：Bug 修复（如 v2.79.1 → v2.79.2）
```

修改位置：`client/src/lib/version.ts`（唯一源头）
