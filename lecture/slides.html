Part 1 — 2024 · The Core Concept
What is
Chatbot Customisation?
Generic AI (ChatGPT, Gemini…)
  • Answers anything — broad, general knowledge
  • No knowledge of your course, chapter, or students
  • Open-ended dialogue — no structure, no workflow
  • Students can ask it to write their essay for them
Customised Chatbot
  • Guided by a system prompt — instructions the educator writes once at the backend
  • Knows the specific content, follows a pedagogical sequence
  • Presents a menu of options — structured interaction, not open chat
  • Can refuse off-topic requests entirely
How It Works
1 Educator writes a system prompt — defines the AI's role, knowledge, workflow, and tone. Plain English. No code.
↓
2 Chatbot is deployed — students access via a shared link. They see only the menu; the system prompt is hidden.
↓
3 Students interact — structured guided learning, shaped entirely by the educator's design choices.
The Shift

From consuming AI → to designing AI behaviour. The educator becomes the architect.

Part 1 — 2024 · Case Study
Programming a Chatbot
in Plain English
The wordchoice bot — poe.com/wordchoice
1Learn about word choice & collocation — 3 MC questions, one at a time, with feedback
2Check collocation of a word — appropriate & inappropriate combinations listed
3Check if a word combination is appropriate — verdict + better alternative
4Review word choices in a paragraph — sentence by sentence, structured feedback
Output template (Option 4)

word combination: word 1 (pos) + word 2 (pos)
verdict: appropriate / not appropriate
explanation & example: …
confidence level: 10%–100%

The system prompt — written entirely in plain English
// Role You are an experienced English lecturer. Help students improve their word choice and collocation. // Menu trigger When user types *menu*, show the 4 options ONLY. Do not add any extra text. // Option 1 — quiz Ask 3 multiple-choice questions one at a time. Wait for the answer before showing the next question. Give brief feedback after each answer. // Option 4 — paragraph review Ask the student to paste their paragraph. Review sentence by sentence. For each sentence, check verb-object combinations and key word choices. Use this template for every combination found: word combination · verdict · explanation · confidence % // Safety — always include at the end "I am powered by AI and can make mistakes. Please consult your teacher for verification."
Natural Language Programming

No code is written. The educator programs the AI's behaviour entirely in plain English — specifying roles, workflows, output formats, and guardrails. This is orchestration, not conversation.

A Precursor of Vibe Coding

Vibe coding asks AI to write programming code from a description — more powerful. But this came first: using natural language to direct AI behaviour directly, without generating code at all.

Part 1 — 2024 · Two Pathways
Customising Chatbots:
Platform-Based vs API-Based
Pathway 1 — Platform-Based (Low-Code)
  • Tools like Poe.com, Character.AI, custom GPTs
  • Write a system prompt in plain English → publish a chatbot link
  • Quick to set up — no programming knowledge needed
  • Free or low cost, good for experimenting
✓ Accessible ✓ No code ✗ No student data ✗ Black box
Pathway 2 — API-Based (Bespoke Platform)
  • Custom platform built on provider APIs (OpenAI, Gemini…)
  • Unlimited system prompt — encode full pedagogical logic
  • Full control: UX, data storage, student access
  • Complete visibility into student conversation logs
✓ Full data ✓ No limits ✗ Needs developer ✗ Higher cost
Where We Were in 2024

Pathway 1 was accessible but left educators blind to their students' learning. Pathway 2 solved that — but required a programmer and months of development. Fast forward two years.

Part 2 — 2026 · Institutional Integration
SCORM & xAPI:
Connecting to Moodle
SCORM — "Did the student do it?"
  • Packages an entire lesson as a portable ZIP module — upload once to any LMS
  • Moodle launches it in an iframe; the module reports back a completed or passed signal
  • Gradebook records: completed ✓ / score / time spent
  • Simple, widely supported, works offline
✓ Any LMS ✓ Zero setup ✗ No conversation data
xAPI — "How did the student learn?"
  • Every interaction fires a structured statement to a Learning Record Store (LRS)
  • Statements follow the pattern: Actor → Verb → Object
  • Captures: questions asked, quiz attempts, time on task, struggle points
  • Moodle reads from the LRS via the Logstore xAPI plugin
// Example xAPI statement
Simon → asked → "What is a system prompt?"
Simon → answered-correctly → Quiz Q3 (score: 80%)
Simon → struggled-with → BYOK setup
Key Distinction

SCORM answers "did they complete it?"  ·  xAPI answers "how did they actually learn?" — the difference between a checkbox and a learning dataset.
